# Scraping Networks

A small pipeline for enriching a list of people — drawn from a network (a company
alumni group, a Slack, a portfolio) — with:

- **LinkedIn** profile URL
- **Current affiliation** (where they are now)
- **Founded company** — only if they actually founded one, as `company url // year`
- **Personal website**

The point isn't raw coverage — it's **precision under same-name ambiguity**.
Every field is found by web search and then *verified*; the tool abstains (leaves
a cell blank) rather than guess, and every filled cell carries a confidence tag
and a one-line evidence note so the uncertain ones are easy to hand-check.

It does **not** scrape LinkedIn. It queries licensed search APIs (Claude's
web-search tool) and verifies what it finds.

## The accuracy trick: link-back verification

Once a LinkedIn URL is found for someone, a candidate personal site that links
*back* to that exact profile is almost certainly theirs. That check is
deterministic — anchored to a unique profile ID, run locally in Python — so
those matches are tagged `linkback` and can be trusted blindly. Everything else
gets a graded confidence (`high`/`medium`/`low`/`none`) and the low/medium ones
go through an adversarial second pass.

## Pipeline

```
  clean_sheet.py      drop test rows, normalize LinkedIn URLs, fix misfiled data
        │
  extract_roster.py   (only if your source is a Slack screen-capture PDF)
        │             pull the page images so you can transcribe names → CSV
        ▼
  enrich.py           per person: web-search → find 4 fields → link-back verify
        │             → fill blanks only, with confidence + evidence columns
        ▼
  verify.py           re-check every low/medium cell with a skeptic prompt;
                      confirm / correct / clear (catches same-name mismatches)
```

| Script | What it does | Needs a key? |
|---|---|---|
| `clean_sheet.py` | Deterministic clean-up (test rows, URL normalization, misfiled values) | no |
| `extract_roster.py` | Extract member-list images from a Slack screen-capture PDF | no |
| `enrich.py` | All-in-one enrichment (LinkedIn + affiliation + founded + website), resumable | `ANTHROPIC_API_KEY` |
| `verify.py` | Adversarial spot-check of the low/medium-confidence cells | `ANTHROPIC_API_KEY` |
| `workflows/` | The Claude Code workflows that generated the shipped CSVs (parallel, at scale) | Claude Code |

`enrich.py` / `verify.py` are the single-process path you can run with one API
key. The data in this repo was generated by the **`workflows/`** versions, which
fan the identical prompts across hundreds of agents in parallel under Claude
Code — see `workflows/README.md`.

## Setup

```bash
python3 -m venv venv && source venv/bin/activate
pip install -r requirements.txt
export ANTHROPIC_API_KEY="sk-ant-..."
```

`enrich.py`/`verify.py` use Claude's server-side **web-search tool**, so the only
key needed is `ANTHROPIC_API_KEY`. By default they use `claude-opus-4-8`; for a
cheap bulk run over hundreds of people set `ANTHROPIC_MODEL=claude-haiku-4-5`.

## Usage

```bash
# 1. (optional) clean the raw sheet
INPUT_CSV="people.csv" OUTPUT_CSV="people_cleaned.csv" ORG_NAME="Y Combinator" \
  python3 clean_sheet.py

# 2. (optional) if your source is a Slack screen-capture PDF, get the images,
#    then transcribe the names into a CSV with at least a `Name` column
python3 extract_roster.py "screencapture-....pdf" roster_imgs/

# 3. enrich — fills blanks only; resumable (re-run to continue after a stop)
INPUT_CSV="people_cleaned.csv" OUTPUT_CSV="people_enriched.csv" \
  CONTEXT_SUFFIX="Y Combinator" python3 enrich.py

# 4. spot-check the uncertain cells in place
CSV="people_enriched.csv" CONTEXT_SUFFIX="Y Combinator" python3 verify.py
```

`CONTEXT_SUFFIX` is the network that pins the right person (`"Y Combinator"`,
`"Square / Block"`, …). Input needs a `Name` column; a free-text hint column
(role, tenure, Slack status — set via `CONTEXT_COLUMN`, default `Past Affiliation`)
sharply improves disambiguation.

## Output columns

`enrich.py` fills these (only when blank — your manual data is never overwritten):

| Column | Meaning |
|---|---|
| `Current Affiliation` | where they are now |
| `Company (if any) // status` | **founded** company as `url // year` (blank unless they founded one) |
| `LinkedIn` | canonical `https://www.linkedin.com/in/<slug>/` |
| `Personal website` | their own site (blank is common and usually correct) |
| `linkedin_conf` / `website_conf` / `founded_conf` | per-cell confidence |
| `enrich_evidence` | one-line note: how identity was confirmed + any same-name flag |

Read the confidence columns as a trust signal:

- **`linkback`** — site links to the known LinkedIn. Essentially certain.
- **`high` / `medium` / `low` / `none`** — the model's own confidence; `low`/`none` usually sit next to a deliberately-blank cell (it looked and abstained).
- After `verify.py`: **`high (verified)`** (re-confirmed), **`corrected`** (value fixed), **`refuted->blank`** (wrong match, cleared), **`uncertain`** (couldn't confirm — eyeball it).

Sort by these to triage; delete the four audit columns once you're satisfied.

## Notes & limitations

- **Blanks are normal and usually correct** — most people have no personal site, and abstaining on a same-name is the right call.
- **~100% is not the goal.** Optimize for precision + a confidence score; the few uncertain rows are flagged for review.
- **Same-name ambiguity is the real risk.** `verify.py` re-checks the low/medium cells, but high-confidence cells aren't exhaustively re-verified — skim the `founded`/`LinkedIn` columns where a result surprises you.
- **Data can be stale** — search lags reality, so a recent job change or fundraise may not show up yet.

## Responsible use

Queries licensed search APIs and public profile data; does not scrape LinkedIn or
evade access controls. The data files (`*.csv`, `*.json`) are **gitignored** — they
hold real people's personal data; don't commit them to a public repo.
